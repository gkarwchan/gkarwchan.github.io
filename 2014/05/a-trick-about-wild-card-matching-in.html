A trick about wild card matching in Linux

May 6, 2014 • Ghassan Karwchan

So, when we do this command : 
echo any data
the result will be:
any data
But what about this command? 
echo p*
the result will be echo the file names which their names start with the letter p.
So, if there are two files with names:
  • pilot1.txt
  • pilot2.txt
Then the result of the previous command will be interpreted as follows: 
echo pilot1.txt pilot2.txt
The reason is the file substitution is occurring on the shell level , and not the command level.
Which means when the shell find the wild-card letters, it will replace them with the matched file names.
Another trick: let us suppose that we want to list the files start with "readme" and by mistake we run the following command: 
ls readme. *
We entered a space in the file name.
Linux will interpret it as list of two things:
First, we are trying to list a file that its name "readme."
and then, the shell will substitute the star with all the file names.
Another consequence of that is the Unix's Find command.
When we run the find command and we want to look for a file that its name start with p for example, we should submit the command as follows: 
find . -name "p*" -print
Notice how we add the quotations around the name criteria.
Without the quotation marks the shell will try to substitute the wildcards with files, and pass them to the find, which in case there is more than one file that start with "p", it will pass all these file names to find which result with a syntax error.
